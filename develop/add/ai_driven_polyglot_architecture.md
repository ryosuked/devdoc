# AIエージェント駆動開発におけるマルチランゲージ・アーキテクチャ設計指針

## 1. 目的
本ドキュメントは、ClineやCursorといったAIエージェントの能力を最大限に引き出し、開発効率とシステム品質を両立させるためのアーキテクチャ構成を定義する。
静的型付け言語（Go）による堅牢性と、動的型付け言語（Ruby）による柔軟なプロトタイピングを組み合わせ、AIとの共生を前提とした「疎結合かつ高凝集」なシステム構築を目指す。

---

## 2. アーキテクチャ構成と役割分担

### 2.1 API Gateway / Edge レイヤー (Go)
*   **AIとの相性:** 言語仕様のシンプルさと標準ライブラリの充実により、AIが生成するコードの揺れが少なく、高精度な実装が可能。
*   **主な責務:**
    *   認証・認可 (JWT検証等)
    *   レート制限、リクエストバリデーション
    *   プロトコル変換 (REST ↔ gRPC)
    *   分散トレーシングのコンテキスト注入

### 2.2 アプリケーション層 / モノリス・コア (Ruby on Rails)
*   **AIとの相性:** 記述が宣言的で簡潔。AIに対して「何をしたいか」を伝えるだけで、複雑なドメインロジックを極めて高い生産性でコード化できる。
*   **主な責務:**
    *   複雑なビジネスロジックの実装
    *   柔軟なデータモデリングとRDB操作
    *   外部サービス連携 (Stripe, SendGrid等)
    *   管理画面およびAPIエンドポイントの提供

### 2.3 サービス層 / マイクロサービス (Go)
*   **AIとの相性:** 責務を単一に絞ることで、AIが把握すべきコンテキスト（ファイル数やロジック量）を小さく保つことができ、バグ混入率を劇的に低下させられる。
*   **主な責務:**
    *   計算負荷の高い特定の処理
    *   高い信頼性が求められるトランザクション処理
    *   独立したスケーリングが必要なコンポーネント

---

## 3. AI駆動開発を成功させる3つの柱

### 3.1 Schema-First Development (IDLの厳格化)
AIが言語の境界を越えて正しく通信コードを生成するため、**gRPC (.proto)** または **OpenAPI (openapi.yaml)** を真実の口（Single Source of Truth）とする。
*   **ワークフロー:** 人間がスキーマを定義 → AIが各言語のボイラープレート（Client/Server）を自動生成。
*   **利点:** AIが「型」を認識できるため、通信エラーのリスクを最小化できる。

### 3.2 コンテキストの物理的隔離 (Directory/Repository Structure)
AIエージェントが「今どこを修正すべきか」を迷わないよう、境界を明確にする。
*   **推奨構造:**
    ```text
    /schema          # .proto や openapi.yaml
    /services
      /gateway       # Go: API Gateway
      /core-app      # Ruby: Main Application
      /payment-svc   # Go: Payment Microservice
    /shared          # 共通の型定義やスクリプト
    ```

### 3.3 分散トレーシングとログの標準化 (OpenTelemetry)
AIに「エラーの原因を特定させる」ために、OpenTelemetryを全サービスに導入する。
*   **実装:** Trace IDをリクエスト間で引き回し、ログを構造化（JSON）する。
*   **AI活用:** AIに特定の Trace ID に紐づくログを解析させることで、複数サービスを跨ぐ複雑なバグの特定時間を短縮する。

---

## 4. 課題と対策

### 4.1 Ruby側の型安全性 (Type Safety)
*   **課題:** Rubyは動的型付けのため、Goとの通信において型不一致が起きやすい。
*   **対策:** 
    *   **Sorbet / RBS** の導入による静的解析の強化。
    *   AIに対し「OpenAPI定義からRubyのバリデーションコードを生成せよ」というプロンプトをテンプレート化する。

### 4.2 インフラ・デプロイの自動化
*   **方針:** TerraformやAnsibleを用いた IaC (Infrastructure as Code) を徹底する。
*   **AI活用:** インフラ定義もAIに記述させる。特にCI/CD（GitHub Actions等）の設定ファイルは、AIが得意とする「定型パターンの適用」領域であるため、積極的に任せる。

---

## 5. 開発フロー案
1.  **要件定義:** 人間が目的をAIに伝える。
2.  **スキーマ設計:** AIが `openapi.yaml` または `.proto` をドラフトし、人間がレビュー。
3.  **コード生成:** AIがスキーマからGo/Rubyそれぞれのコードを生成。
4.  **ロジック実装:** AIがビジネスロジックを実装し、テストコード（RSpec / go test）を併記。
5.  **検証:** CIでテストと型チェックをパスすることを確認。

---

## 6. 結論
このマルチランゲージ構成は、単なる技術的な適材適所ではなく、**「AIエージェントが最も力を発揮しやすい環境」**を構築することを目的としている。
Goで「枠組み」を固め、Rubyで「速度」を出し、スキーマで「結合」を担保することで、少人数かつ高速な開発サイクルを実現する。
