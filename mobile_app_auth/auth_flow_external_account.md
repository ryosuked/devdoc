# モバイルアプリ認証フロー：外部連携・ID/PASS認証

ユーザーが自社サイトのID/パスワードを入力したり、Google/Appleなどの外部アカウント（IdP）と連携したりして認証を行うフローです。OIDC/OAuth2 に基づく標準的な手法です。

---

## 1. 概要
モバイルアプリにおいて、安全に外部認証を行うためには、アプリ内に直接パスワードを入力させるのではなく、**システムブラウザ（ASWebAuthenticationSession 等）**を介した認可コードフローを用いることが強く推奨されます。

## 2. 認証シーケンス（OAuth 2.0 + PKCE）

1.  **ログインアクション**: ユーザーが「ログイン」ボタンをタップ。
2.  **ブラウザ起動**:
    *   アプリが `Code Verifier` と `Code Challenge` を生成（PKCE）。
    *   外部の認可サーバー（Auth0, Firebase Auth, 自社認可サーバー等）のURLを、システムブラウザで開く。
3.  **ユーザー認証**:
    *   ブラウザ上でユーザーがID/PASSを入力、またはSNSボタンを選択。
    *   認可サーバーが本人を確認。
4.  **認可コードの返却**:
    *   サーバーがアプリ固有のカスタムURLスキーム（`my-app://callback?code=xxx`）へリダイレクト。
    *   アプリがこの「認可コード」をキャッチ。
5.  **トークン交換**:
    *   アプリがバックエンドのトークンエンドポイントに対し、「認可コード」と「Code Verifier」を送信。
    *   サーバーは検証後、**IDトークン**と**アクセストークン**、**リフレッシュトークン**を返却。
6.  **セッション開始**:
    *   アプリはトークンをセキュアに保存し、ログイン状態となる。

## 3. アカウント連携のパターン

### A. ソーシャルログイン (Social Auth)
*   Google, Apple, Facebook等の既存アカウントを利用。
*   ユーザーは新規にパスワードを覚える必要がなく、信頼性が高い。

### B. ID/パスワード (Custom Identity)
*   自社でユーザーDBを管理。
*   メール認証や多要素認証（MFA）を組み合わせてセキュリティを高めることが一般的。

## 4. 特徴と考慮事項

*   **セキュリティ**: アプリがユーザーの生パスワードに触れないため、安全性が高い（OAuth2のメリット）。
*   **ポータビリティ**: 機種変更時も、同じ外部アカウントでログインすればデータを引き継げる。
*   **UX**: 初回のログイン操作が必要になるため、匿名認証に比べるとステップ数が増える。

---

## 5. まとめ
| 項目 | 外部連携・ID/PASS |
| :--- | :--- |
| **信頼性** | 高い（本人確認が明確） |
| **データ保持** | 恒久的（機種変更に強い） |
| **推奨技術** | OAuth 2.0 + PKCE, OIDC |
| **実装難易度** | 端末ID割り当てに比べると高い |
