# OpenID Connect における OP と UserInfo Endpoint の分離について

OpenID Connect (OIDC) のアーキテクチャにおいて、OpenID Provider (OP) と UserInfo Endpoint は論理的に異なる役割を持っており、これらを同一のシステムに置くか、別のサービスとして分離するかは設計上の選択肢となります。

## 1. 論理的な役割の違い
OIDC の仕様上、これらは以下のように分類されます。
- **OpenID Provider (OP):** 認可サーバー (Authorization Server)。ユーザー認証を行い、トークン（ID/Access）を発行する。
- **UserInfo Endpoint:** リソースサーバー (Resource Server)。アクセストークンを受け取り、ユーザー属性情報を返す。

## 2. 同一システム内に置く構成（一般的）
多くの IdP 製品（Auth0, Keycloak, Okta など）やクラウドサービス（Google, Azure AD）では、これらは一つのパッケージとして提供されています。

### メリット
- **実装の簡素化:** トークンの生成・検証ロジックやユーザーDBを共有できるため、設定が容易です。
- **パフォーマンス:** 内部的な通信やデータベース参照で完結するため、レスポンスが高速です。
- **整合性:** 認証状態と属性情報の同期が容易です。

## 3. 別サービスとして分離する構成（高度な設計）
マイクロサービスアーキテクチャや、既存の基幹システムがある場合に採用されます。

### 分離するケースの例
- **データの所在:** 認可機能は外部製品を使うが、ユーザーのプロファイル詳細は自社の基幹データベースにある場合。
- **専門化:** 認可（ログイン処理）とプロファイル管理（属性編集や参照）をサービスとして独立させ、個別にスケールさせたい場合。

### 分離する際の必須条件
1. **トークンの検証手段:** UserInfo 側が、OP の発行したアクセストークンの正当性を検証できる必要があります（公開鍵 JWT 検証や、Token Introspection の利用）。
2. **メタデータの公開:** `/.well-known/openid-configuration` にて、UserInfo Endpoint の URL を正しく RP（クライアント）に通知する必要があります。

## 4. 結論
**物理的な配置はどちらでも問題ありません。**

- **推奨:** 特段の理由がない限り、IdP が提供する標準の UserInfo Endpoint を使用するのが最も安全で低コストです。
- **選択肢:** ユーザー情報の管理を既存の API や別システムに集約したい場合は、その API を OIDC 準拠の UserInfo Endpoint として振る舞わせる（アクセストークンを受け取れるようにする）設計が可能です。
