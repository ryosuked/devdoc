# OpenID Connect (OIDC) 構成要素の解説

OpenID Connect (OIDC) は OAuth 2.0 を基盤とした認証プロトコルであり、以下の主要な構成要素で成り立っています。

### 1. OpenID Provider (OP) / アイデンティティ・プロバイダ (IdP)
ユーザーの認証を行い、認証結果（IDトークン）と認可情報（アクセストークン）を発行するサーバーです。
- **役割:** ユーザーが「誰であるか」を検証し、その情報をクライアントに安全に伝えます。
- **代表例:** Google, Microsoft Entra ID (旧 Azure AD), Auth0, Keycloak など。

> **【補足】OP と IdP の違い**
> - **IdP (Identity Provider):** ユーザーの身元を管理し認証を提供するシステムの総称（広義の概念）。SAMLなど他のプロトコルも含む。
> - **OP (OpenID Provider):** OIDC の仕様に準拠して ID トークンを発行するサーバーを指す（プロトコル用語）。
> 実務上はほぼ同義として扱われますが、仕様上の正確な呼称は OP です。

### 2. Relying Party (RP) / クライアント (Client)
ユーザーの認証を OP に委ね、その結果を利用するアプリケーションです。
- **役割:** 自ら認証機能を実装する代わりに OP を信頼（Rely）し、ログイン処理を行います。
- **形態:** Webアプリケーション、ネイティブアプリ（iOS/Android）、SPA など。

### 3. End User / ユーザー
サービスを利用し、認証を受ける主体となる個人です。
- **役割:** OP に対して自身のアイデンティティを証明（ログイン）し、クライアントへの情報提供を承認します。

### 4. ID Token (IDトークン)
ユーザーの認証情報を格納した、JWT (JSON Web Token) 形式の署名付きデータです。
- **内容:** ユーザー識別子 (sub)、発行者 (iss)、発行日時 (iat)、有効期限 (exp) などが含まれ、クライアント側でユーザー確認に使用されます。

### 5. Access Token (アクセストークン)
保護されたリソース（APIなど）へアクセスするための「鍵」として機能する文字列（またはJWT）です。
- **役割:** クライアントがユーザーの権限を代行してリソースにアクセスすることを許可します。

### 6. Resource Server (リソースサーバー)
ユーザーの属性情報（住所、プロフィールなど）を保持し、有効なアクセストークンを提示したクライアントにデータを提供するサーバーです。
- **UserInfo Endpoint:** OIDC で定義されている、ユーザーの詳細なプロフィールを返すエンドポイントもこれに含まれます。

### 7. Authorization Server (認可サーバー)
トークンの発行、認可フローの管理を行うサーバーです。
- **備考:** OIDC においては OpenID Provider (OP) がこの役割を兼ねています。
