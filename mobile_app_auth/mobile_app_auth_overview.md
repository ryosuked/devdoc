# モバイルアプリにおける認証・認可基盤の概要

モバイルアプリのバックエンド連携において一般的に利用される OAuth 2.0、OpenID Connect (OIDC)、および JWT (JSON Web Token) の役割と、それらの関連性について解説します。

---

## 1. 各技術の役割

### OAuth 2.0 (認可フレームワーク)
*   **役割**: 「認可（Authorization）」のための仕組み。
*   **目的**: ユーザーが、自分の持っているリソース（例：プロファイル情報、写真など）に対して、サードパーティアプリがアクセスすることを許可するために使用されます。
*   **成果物**: **アクセストークン (Access Token)**。これは「何ができるか（スコープ）」を示す鍵です。
*   **注意**: 本来は「認証（誰であるか）」のためのプロトコルではありません。

### OpenID Connect / OIDC (認証レイヤー)
*   **役割**: 「認証（Authentication）」のための仕組み。
*   **目的**: OAuth 2.0 の仕組みを拡張し、ユーザーが「誰であるか」を特定するための情報を取得可能にします。
*   **成果物**: **IDトークン (ID Token)**。ユーザー属性情報を含むトークンです。
*   **関係**: OAuth 2.0 の上に構築されたアイデンティティレイヤーです。

### JWT / JSON Web Token (データ形式)
*   **役割**: 情報を安全に転送するための「データフォーマット」。
*   **目的**: トークン自体に署名を付与し、改ざんを検知可能にすることで、ステートレスな情報の受け渡しを実現します。
*   **特徴**: ヘッダー、ペイロード、署名の3つのセクションで構成されます。
*   **関係**: OIDC の IDトークンは、実体として JWT 形式で提供されます。また、OAuth 2.0 のアクセストークンも JWT として実装されることが一般的です。

---

## 2. 技術同士の関連図

これらの技術は、以下のような包含・利用関係にあります。

```text
+-----------------------------------------------------------+
| OpenID Connect (認証レイヤー)                               |
|   +-----------------------------------------------------+ |
|   | OAuth 2.0 (認可フレームワーク)                        | |
|   |   +---------------------------------------------+   | |
|   |   | 利用されるトークン形式: JWT (JSON Web Token)  |   | |
|   |   +---------------------------------------------+   | |
|   +-----------------------------------------------------+ |
+-----------------------------------------------------------+
```

1.  **OAuth 2.0** がフロー（認可コードフローなど）を定義します。
2.  **OIDC** がそのフローの中で「IDトークン」を発行する仕組みを追加します。
3.  **JWT** が「IDトークン」や「アクセストークン」の具体的なデータ構造として採用されます。

---

## 3. モバイルアプリでの標準的なワークフロー

1.  **ログイン開始**: アプリがブラウザ（ASWebAuthenticationSession 等）を開き、認可サーバーへリダイレクト。
2.  **認証と承認**: ユーザーがログインし、アプリへの権限付与を承認。
3.  **認可コードの受け取り**: アプリが「認可コード」を受け取る（PKCEの使用が推奨）。
4.  **トークン交換**: アプリが認可コードをサーバーへ送り、以下を取得。
    *   **IDトークン (JWT)**: ユーザー名の表示などに使用。
    *   **アクセストークン (JWT等)**: APIリクエストの `Authorization` ヘッダーに付加。
    *   **リフレッシュトークン**: アクセストークンの有効期限が切れた際の再発行に使用。
5.  **APIアクセス**: アクセストークンを使ってバックエンドAPIを呼び出す。

---

## 4. まとめ

| 技術 | カテゴリ | 役割 | 成果物 |
| :--- | :--- | :--- | :--- |
| **OAuth 2.0** | フレームワーク | アクセス権限の委譲（認可） | アクセストークン |
| **OIDC** | プロトコル | 本人確認の仕組み（認証） | IDトークン |
| **JWT** | フォーマット | データの持ち運び・改ざん検知 | (トークンの実体) |

モバイルアプリ開発においては、**OIDC** のフローに従って **OAuth 2.0** の認可コードフロー（+PKCE）を実行し、取得した **JWT** 形式のトークンでセッション管理やAPI連携を行うのが現代のベストプラクティスです。
